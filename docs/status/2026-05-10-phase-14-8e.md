# Phase 14.8e Tier 5 — Production-realism examples shipped (2026-05-10)

**Last updated**: 2026-05-10 (Phase 14.8e Tier 5 — production-
realism example library shipped. Three examples cover deployment
shapes that the underlying packages support but that the Tier 1–4
examples deliberately stayed away from to keep their focus narrow:

1. `e-commerce-orders` — flagship. Three bounded contexts (Orders /
   Inventory / Billing) on three Postgres DataSources, each with
   its own outbox stack. Order saga choreographed through outbox
   integration events: `OrderPlacedEvent → StockReserved → PaymentCharged →
   OrderConfirmedEvent`. Externalization to Kafka on the terminal
   event only (`@Externalized` decorator + `OutboxMicroservicesModule`).
   CQRS `@CommandHandler` / `@QueryHandler` for write/read split, with
   `aggregate.commit()` flowing through `HybridEventPublisher` to
   both the in-memory dispatcher and outbox. REST surface via
   `OrdersController`. Compensation handlers on both failure
   branches (`StockReservationFailedEvent`,
   `PaymentFailedEvent` → release reserved stock atomically with
   marking the order failed). Idempotency gates everywhere
   (reservations PK `${orderId}:${sku}`, payments PK `orderId`,
   confirmations gated on `status='placed'`). 8 integration tests
   via testcontainers Postgres × 3 databases + mocked Kafka client.
   2031 LoC.
2. `async-config-from-environment` — `forRootAsync` end-to-end on
   three of four framework modules (`TypeOrmModule`, `OutboxModule`,
   `OutboxTypeOrmModule`); `TypeOrmTransactionalModule` falls back
   to sync `forRoot()` because of a framework bug surfaced by this
   example (see Convention #22 below). `ConfigModule.forRoot` with
   Joi validation (`abortEarly: false` for one-pass diagnosis) and
   three `.env.{development,staging,production}` profiles. Outbox
   tunables (`pollingInterval`, `batchSize`, `maxConcurrent`) flow
   through `ConfigService` into `OUTBOX_PROCESSOR_OPTIONS`. 5
   integration tests: dev boot atomicity + dev tunables-injection +
   prod tunables-injection (asserts DIFFERENT values reach the
   processor) + missing-env-rejected + out-of-range-rejected.
   testcontainers Postgres + side `pg.Client` for fixture-env tests
   that don't need DB. 983 LoC.
3. `graceful-shutdown` — `app.enableShutdownHooks()` plus a
   user-side `OutboxDrainService` (implements
   `OnApplicationShutdown`, idempotently re-calls `processor.stop()`,
   polls `findIncomplete()` until no row is in `PROCESSING` state
   or `DRAIN_TIMEOUT_MS` elapses). Plugs the gap in the framework's
   `OutboxProcessingModule.onApplicationShutdown` (sync, does NOT
   await the in-flight `processBatch()` Promise). Plus an
   `ExampleCleanupService` stand-in for arbitrary user
   `OnApplicationShutdown` cleanup (metrics flush, cache close,
   etc.). 4 integration tests: idle-close + mid-handler-completion
   + mid-tx-atomicity + user-hook-fires. testcontainers Postgres +
   side `pg.Client` for post-close verification. 868 LoC.

Shipped in 4 commits (3 example commits + this closure). 17 tests
(0 unit + 17 testcontainers integration across the three examples).
591/591 package baseline preserved across all four commits.

## New conventions surfaced

Seven new conventions inscribed in
[`docs/status/conventions.md`](conventions.md):

- **Convention #18 — Inner-method indirection for
  `@Transactional({ dataSource: 'X' })` inside an
  `@IntegrationEventsHandler`.** The cqrs scanner captures
  `instance.handle.bind(instance)` in `OnModuleInit`; the
  `TransactionalMethodsBootstrap` (`OnApplicationBootstrap`) wraps
  methods AFTER. Naive `@Transactional({ dataSource: 'inventory' })`
  on the public `handle()` method does NOT take effect — the bound
  reference is the un-wrapped original. Workaround: `handle()` (no
  decorator) delegates to a private `processInInventoryTx()`
  (decorated). The `this.method()` call resolves at invocation time
  — by then the bootstrap has installed the wrapped version on the
  instance. See `examples/e-commerce-orders/src/inventory/release-stock.handler.ts`
  and `billing/charge-payment.handler.ts` for the canonical pattern.
  Single-DS handlers (where the worker's outer `REQUIRES_NEW`
  transaction is on the same DS as the listener target) don't need
  the indirection — the outer tx already covers the work.

- **Convention #19 — `@Externalized` events still need a local
  `@OutboxEventsHandler` listener to materialise a publication
  row.** Convention #15 (silent no-op without listener) extends to
  externalization: the externalizer pipeline reads from
  `event_publication`, but the publication is only created when at
  least one listener is registered for the event class. The
  `@Externalized` decorator alone is not sufficient. Pattern: a
  stub `@OutboxEventsHandler` whose body is empty (or carries an
  audit-trail / read-model-update side effect). See
  `examples/e-commerce-orders/src/orders/externalized-event-stub.ts`
  for the canonical empty-stub form.

- **Convention #20 — `CqrsTransactionalModule` does NOT export
  `CommandBus` / `QueryBus` to consumers.** The module imports
  `CqrsModule` internally and overrides `EventPublisher`
  (Convention #6). A duplicate `CqrsModule.forRoot()` import in the
  consumer would shadow the override; consequently
  `CqrsTransactionalModule` deliberately scopes `CqrsModule`'s
  exports inward. REST controllers and other non-handler consumers
  cannot inject `CommandBus.execute(...)` — they must inject the
  command and query handlers directly and call `handler.execute(...)`
  / `handler.handle(...)`. See
  `examples/e-commerce-orders/src/orders/orders.controller.ts` for
  the pattern.

- **Convention #21 — `OutboxModule.forRootAsync({ repository })`
  takes `repository` at the OPTIONS level, NOT in the async factory
  result.** Provider tokens must be declared at module-build time,
  so the module reads `repository` (and `serializer`) from the
  options argument synchronously. The async factory's return shape
  (`OutboxModuleAsyncFactoryResult`) only carries *runtime
  tunables* — `processor`, `staleness`, `republishOnStartup`,
  `startupBatchSize`, `completionMode`. Putting `repository` inside
  `useFactory`'s return is silently ignored; the module falls back
  to `InMemoryEventPublicationRepository`, the publication never
  reaches Postgres, and the worker never delivers anything. The
  silence is double-layered: the misplaced field is dropped without
  a warning, AND `OutboxEventPublisher.publish` is then a silent
  no-op for events that have no listener wiring on the (in-memory)
  default — Convention #15 again. Surfaced 2026-05-10 in the
  Phase 14.8e async-config example's first integration run; the
  audit row committed but the publication count was zero. See
  `examples/async-config-from-environment/src/app.module.ts` for the
  correct top-level `repository:` placement.

- **Convention #22 — `TypeOrmTransactionalModule.forRootAsync` has
  a bootstrap bug when combined with `TypeOrmModule.forRootAsync`.**
  TypeORM's `PostgresDriver.createPool` raises
  `TypeError: this.postgres.Pool is not a constructor` and stalls
  in the `@nestjs/typeorm` connect-retry loop. The error happens
  inside TypeORM's own DataSource init, BEFORE the registration
  factory of `TypeOrmTransactionalModule.forRootAsync` runs.
  Bisected: the bug is triggered solely by adding the async-variant
  module to imports — sync `forRoot()` of the same module works
  cleanly in the same composition.

  **Workaround**: call `TypeOrmTransactionalModule.forRoot()` (sync)
  even when the rest of the stack is `forRootAsync`. The module
  has no async-resolvable tunables anyway (`dataSource` and
  `isDefault` are statically declared per the JSDoc on
  `TypeOrmTransactionalAsyncOptions`), so there is no semantic
  loss. The async-config example does exactly that. Tracked as a
  framework-fix task ahead of broader Tier 5 promotion of
  `forRootAsync`. Cross-referenced in
  [`docs/known-limitations.md`](../known-limitations.md).

- **Convention #23 — dotenv refuses to overwrite an existing
  `process.env` key.** Two consequences: (1) integration tests
  that load multiple `.env` files sequentially in the same Jest
  worker get cross-contamination — the FIRST file's values mask
  later files' values for the duration of the worker, breaking any
  test that depends on profile-switching. The async-config example
  snapshots managed env keys at `beforeAll`, restores between
  tests; the helper is small but load-bearing. (2) In production,
  exported shell variables override `.env` files — that's
  *intended* (secrets-manager-injected env should win over a
  committed file), but it surprises operators who expect "set in
  `.env`, gets the value." Don't rely on `.env` to "reset" a
  variable that has already been set in the deployment
  environment.

- **Convention #24 — Outbox graceful drain requires a user-side
  `OnApplicationShutdown` complement.** The framework's
  `OutboxProcessingModule.onApplicationShutdown` calls
  `processor.stop()` synchronously: it sets `running = false` and
  cancels the next-poll `setTimeout`. It does NOT await the
  `processBatch()` Promise that the previous tick already
  dispatched. NestJS proceeds through the rest of the shutdown
  chain and disposes the DataSource provider — at which point an
  in-flight `processOne()` writing `PROCESSING → COMPLETED` can
  race the pool teardown, leaving the row stuck in `PROCESSING`
  until the staleness monitor recovers it on the next boot.

  **Pattern**: a user-side `OutboxDrainService` that implements
  `OnApplicationShutdown` async, idempotently re-calls
  `processor.stop()` (safe regardless of NestJS shutdown ordering),
  then polls `EventPublicationRepository.findIncomplete()` until no
  row is in `PROCESSING` state or `DRAIN_TIMEOUT_MS` elapses
  (10 s default — tune to your platform's grace period). See
  `examples/graceful-shutdown/src/shutdown/outbox-drain.service.ts`
  for the canonical pattern. The framework may absorb this in a
  future phase by making `processor.stop()` await an in-flight-
  batch `Promise`; the user-side complement remains valid as an
  explicit drain envelope under operator control either way.

## Design observations carried over from prior tiers and reinforced

- **Idempotency gate at every cross-context handler.** Tier 3's
  `externalization-with-fallback` introduced the inbox/dedup
  template; Tier 4 saga-pattern and audit-logging applied it at
  the step level. Tier 5's `e-commerce-orders` extends the
  pattern across three context boundaries simultaneously:
  reservations PK `${orderId}:${sku}`, payments PK `orderId`,
  confirmations gated on `status='placed'`. The pattern is now
  consistently applied across six examples.

- **Asymmetric multi-`forRoot` is the natural shape when one DS
  is purely a sink.** Tier 4's `audit-logging` made this explicit.
  Tier 5's `e-commerce-orders` extends to a non-sink-symmetric
  multi-DS deployment: every DS gets the full outbox stack (each
  publishes its own integration events) but only the orders DS
  registers `OrderConfirmedEvent` for externalization. The
  per-DS scope of `OutboxModule.forFeature` and the per-DS
  routing of the smart-facade `OutboxEventPublisher` make the
  asymmetry composable without ceremony.

- **Externalization on the terminal event only.** A multi-step
  saga has many internal events (one per step boundary) and one
  business-meaningful terminal event. Putting `@Externalized` on
  the terminal event only avoids leaking the saga's internal
  step model to downstream consumers — they couple to the
  business outcome, not the implementation. `e-commerce-orders`
  emits `OrderConfirmedEvent` to Kafka; `StockReserved`,
  `PaymentCharged`, etc. stay in-process.

## Audit-estimate variance pattern (Phase 14.8e retrospective)

Tier 4 closure recommended anchoring Tier 5 estimates on Tier 4
exemplars (688-1149 LoC envelope, +1200 ceiling for multi-axis
examples). Tier 5 actuals: 2031 / 983 / 868. The flagship
(`e-commerce-orders`) blew the +1200 ceiling — three contexts ×
three DataSources × outbox saga × Kafka externalization × CQRS ×
REST is genuinely a multi-multi-axis surface, the Tier 4 anchor
underestimated. The other two sit comfortably inside the Tier 4
envelope.

**New envelope guidance for Tier 6+ (if it materialises) or
post-Phase 14.8 examples:**

- Single-axis or two-axis examples: 700-1000 LoC (read-write-
  separation 688 → audit-logging 884 → testing-patterns 896).
- Multi-axis examples (multi-DS + outbox + saga, OR multi-DS +
  externalization): 900-1200 LoC (saga-pattern 1149,
  externalization-multi-datasource 1184, async-config 983).
- Flagship multi-multi-axis (3+ DSes × outbox × CQRS × REST ×
  externalization): 1800-2100 LoC accepted as the realistic floor.
  The audit should still flag deviations beyond +2200 to keep the
  example digestible.

The exceedance on `e-commerce-orders` is justified by the
README: 250+ lines of architecture diagram, bounded-context table,
event ownership/consumption matrix, common pitfalls (three
inscribed as Conventions #18, #19, #20), and cross-references.
The pattern is "production-shape examples earn their LoC by
explaining themselves" — flagship examples should not be lean.

## Framework gaps surfaced

Two real framework asymmetries surfaced; both are documented as
conventions (#22 and #24) with user-land workarounds in the
examples. Background tasks for framework fixes:

- **`TypeOrmTransactionalModule.forRootAsync` Pool-not-constructor
  bug.** Tracked as a separate task — investigation and fix to
  land before any future example promotes the module to async.
  Hypothesis: NestJS provider resolution order under
  `forRootAsync`'s `inject: [asyncToken, ADAPTER_REGISTRY,
  ModuleRef]` (no DS-token dependency) shifts something in
  TypeORM's own init path that the sync-`forRoot` (with
  `inject: [getDataSourceToken(...)]` direct DS dependency)
  doesn't. Bisection notes captured in the spawned task prompt.
  Two `typeorm` package copies in the pnpm store
  (`typeorm@0.3.28_pg…` vs `typeorm@0.3.28_ioredis…`) are a
  candidate root cause worth investigating.

- **`OutboxProcessingModule.onApplicationShutdown` synchronous,
  doesn't await in-flight `processBatch()`.** Not separately
  tracked yet — the user-side `OutboxDrainService` pattern is
  clean and operator-controllable, and there's a credible
  argument that explicit drain envelopes belong in user code
  (different platforms have different grace periods, different
  ops teams want different drain semantics). A framework
  promotion to "await in-flight on stop()" would be a non-
  breaking refinement; deferred until there's user demand or a
  cleaner internal abstraction.
