# Phase 14.8d Tier 4 — Advanced-pattern examples shipped (2026-05-10)

**Last updated**: 2026-05-10 (Phase 14.8d Tier 4 — advanced-pattern
example library shipped. Four examples cover patterns that the
underlying packages support but that did not fit cleanly into the
foundational / multi-DS / externalization tiers:

1. `saga-pattern` — choreographed 4-step saga on a single Postgres
   DS coordinated through the outbox. Steps:
   `OrderPlaced → ReserveInventory → ChargePayment → ShipOrder`.
   Two failure branches: out-of-stock during reservation
   (terminal); authorisation-decline during payment (compensation
   restores the previously-decremented stock). Choreography
   throughout — no orchestrator class. Idempotency gates per step
   via PK `unique_violation` catches (reservation, payment) and
   conditional `UPDATE` predicates (shipment, payment-failure
   compensation). 5 integration tests via testcontainers Postgres
   (happy / payment-failure / reservation-failure / idempotent
   step / atomicity at saga entry). 1149 LoC.
2. `audit-logging` — two physical Postgres DBs wired
   asymmetrically: full outbox stack on **business** DS, only
   `TypeOrmTransactionalModule.forRoot({ dataSource: 'audit' })`
   on the audit DS (sink). `AccountService` writes balance +
   operation row + outbox publication atomically (DD-019);
   `AuditHandler` consumes via outbox and writes the audit row
   under `@Transactional({ dataSource: 'audit' })`. Cross-DS
   distributed transactions intentionally absent (DD-023);
   consistency through at-least-once + idempotent
   `AuditLogRow.operationId` PK gate. 4 integration tests
   (happy / business rollback / idempotent audit / audit-DS down
   recovers). 884 LoC.
3. `read-write-separation` — master and replica `TypeOrmModule.forRoot`
   registrations; **only** master gets `TypeOrmTransactionalModule.forRoot({ isDefault: true })`.
   Replica is `synchronize: false` (mirrors the production
   constraint that read replicas reject DDL). DI graph encodes the
   read/write boundary: `ArticleService` injects
   `@InjectRepository(ArticleRow)` (master); `ArticleQueryService`
   injects `@InjectRepository(ArticleRow, 'replica')`. A misplaced
   `@Transactional({ dataSource: 'replica' })` fails fast at
   bootstrap because no adapter is registered. README documents
   TypeORM's native `replication` option as the alternative
   shape. 4 integration tests (write/read round-trip / write
   rollback / repository-binding DI assertion / cross-session
   isolation). 688 LoC — single-axis simplicity.
4. `testing-patterns` — meta-example with three test tiers
   against the same `WalletService` domain:
   - **Tier 1** unit (`InMemoryTransactionAdapter`, Jest-mocked
     repo) — 4 cases asserting `committedTransactions` /
     `rolledBackTransactions` arrays and business branches.
     Sub-millisecond per case.
   - **Tier 2** outbox unit (`InMemoryEventPublicationRepository`
     auto-default of `OutboxModule.forRoot({})` + `PublishedEvents`
     + `AssertablePublishedEvents`) — 3 cases including the
     rollback-removes-publication invariant pinned by the
     `afterRollback` hook the in-memory repo registers.
   - **Tier 3** integration (testcontainers Postgres) — 3 cases
     with the production `WalletModule`, `@IntegrationEventsHandler`
     listener delivered by the worker, `waitFor` for async
     delivery, publication transitions to `COMPLETED`.
   No `main.ts` (the tests ARE the demo). 896 LoC.

Shipped in 5 commits (4 example commits + this closure). 16 tests

Three new conventions surfaced and inscribed in
`docs/status/conventions.md`:

- **Convention #15 — `OutboxEventPublisher.publish` is a silent
  no-op when no listener is registered for the event type.** The
  `OutboxModule.forFeature([...])` registration is necessary but
  not sufficient: at least one decorated listener
  (`@OutboxEventsHandler` or `@IntegrationEventsHandler`) must
  exist for the outbox to write a publication row. Tests that
  assert on publications must register a listener (a stub class
  is fine). Discovered while writing the testing-patterns
  outbox-unit tier: the first iteration emitted via `publish` and
  asserted via `PublishedEvents` and the assertion failed with
  zero rows.

- **Convention #16 — `@TransactionalEventsHandler` (cqrs in-memory
  dispatcher) does NOT receive events published through
  `OutboxEventPublisher.publish` directly.** The two paths are
  decoupled: the in-memory dispatcher consumes from cqrs's
  `EventBus.publish` / `AggregateRoot.commit()`; the outbox
  consumes from `OutboxEventPublisher.publish`. To bridge the
  paths, either emit through cqrs (which `HybridEventPublisher`
  fans out to both) or use `@IntegrationEventsHandler` (outbox-
  routed when wired). Discovered while writing the testing-patterns
  integration tier: the first iteration used
  `@TransactionalEventsHandler` and the projection's invocations
  array stayed empty.

- **Convention #17 — Subpath imports
  (`@nestjs-transactional/{core,outbox}/testing`) require the
  consuming `tsconfig` to set `module: Node16`,
  `moduleResolution: Node16`, and `isolatedModules: true`.** The
  monorepo `tsconfig.base.json` uses classic `node` module
  resolution which cannot read package.json `exports` subpaths;
  examples that consume the `/testing` exports must override.
  `basic-cqrs` already does this (it imports
  `@nestjs-transactional/core/testing`); other examples not yet
  importing subpaths run fine on the inherited config. The
  conventional override has been documented so future examples do
  not repeat the surprise. A future cleanup may flip the base
  config; out of scope for Tier 4.

Two design observations carried over from prior tiers and
reinforced:

- **Idempotency gate at every outbox-driven step.** Tier 3's
  `externalization-with-fallback` introduced the consumer-side
  inbox/dedup template. Tier 4's `saga-pattern` and
  `audit-logging` each apply the same pattern at the step level:
  primary-key `INSERT` as the gate, `unique_violation` catch as
  the idempotent skip. The pattern is now consistently named and
  documented across three examples.

- **Asymmetric multi-`forRoot` is the natural shape when one DS
  is purely a sink.** `audit-logging` makes this explicit: the
  audit DS gets only `TypeOrmTransactionalModule`, no outbox
  stack. The pattern composes cleanly with ADR-019's
  multi-`forRoot` shape — each DS gets exactly the components it
  needs, no overhead from registering machinery a sink does not
  use.
